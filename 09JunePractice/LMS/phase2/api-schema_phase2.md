# API Schema Overview — Learning Management System (LMS) Feature Enhancements

## 1. Architecture
* **API Style**: RESTful API utilizing JSON format for payload communication and multi-part form-data for binary file uploads.
* **Versioning**: Versioning is managed through URL routing prefixes. All Phase 2 routes are versioned under `/api/`.
* **Base URL**: `https://api.yourapp.com/api` (Production gateway) or `http://localhost:5000/api` (Local development environment).
* **Authentication**: Stateless tokens utilizing standard HTTP Bearer Authentication. JWT tokens are verified on the backend via the `authMiddleware.js` verification pipeline.

## 3. Common Schemas

### 1. Success Response Schema
Every successful REST response wraps the result in a standard JSON envelope:
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["success", "data"],
  "properties": {
    "success": {
      "type": "boolean",
      "const": true
    },
    "data": {
      "type": "object",
      "description": "Dynamic resource payload returned by the specific endpoint."
    },
    "message": {
      "type": "string",
      "description": "Optional user-friendly execution summary text."
    }
  }
}
```

### 2. RFC7807 Error Schema
API operational and runtime errors conform to the RFC7807 problem details specification:
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["success", "type", "title", "status", "detail", "instance"],
  "properties": {
    "success": {
      "type": "boolean",
      "const": false
    },
    "type": {
      "type": "string",
      "format": "uri",
      "description": "A URI reference that identifies the specific error type."
    },
    "title": {
      "type": "string",
      "description": "A short, human-readable summary of the problem type."
    },
    "status": {
      "type": "integer",
      "description": "The HTTP status code generated by the origin server."
    },
    "detail": {
      "type": "string",
      "description": "A human-readable explanation specific to this occurrence of the problem."
    },
    "instance": {
      "type": "string",
      "format": "uri",
      "description": "A URI reference that identifies the specific occurrence of the error."
    },
    "errors": {
      "type": "array",
      "description": "Optional list of sub-validation errors (e.g. invalid form fields).",
      "items": {
        "type": "object",
        "required": ["field", "message"],
        "properties": {
          "field": { "type": "string" },
          "message": { "type": "string" }
        }
      }
    }
  }
}
```

### 3. Pagination Query Schema
Query parameters applied on collection lists (e.g. reading doubt lists or badge archives):
```json
{
  "type": "object",
  "properties": {
    "page": {
      "type": "integer",
      "minimum": 1,
      "default": 1
    },
    "limit": {
      "type": "integer",
      "minimum": 1,
      "maximum": 100,
      "default": 20
    }
  }
}
```

### 4. JWT Payload Schema
Format of the verified decrypted payload stored inside the encrypted token:
```json
{
  "type": "object",
  "required": ["id", "role", "iat", "exp"],
  "properties": {
    "id": {
      "type": "string",
      "description": "The unique database Mongoose ObjectID representing the authenticated user."
    },
    "role": {
      "type": "string",
      "enum": ["Learner", "Instructor"]
    },
    "iat": {
      "type": "integer",
      "description": "Timestamp indicating token issuance time."
    },
    "exp": {
      "type": "integer",
      "description": "Timestamp indicating expiration time (valid for 24 hours)."
    }
  }
}
```

### 5. File Upload Payload Schema
Multipart form-data constraints applied on upload paths:
```json
{
  "type": "object",
  "required": ["file"],
  "properties": {
    "file": {
      "type": "string",
      "format": "binary",
      "description": "File binary stream. Enforced size < 10MB. Content types: pdf, zip, doc, docx."
    }
  }
}
```

## 4. HTTP Status Codes
* **200 OK**: Request completed successfully. Returns resource payloads.
* **201 Created**: Resource created successfully. Returns the new object data.
* **400 Bad Request**: Missing or invalid parameters. Returns RFC7807 validation error list.
* **401 Unauthorized**: JWT token is missing, expired, or failed signature checks.
* **403 Forbidden**: Role checks failed (e.g., Learner accessing instructor tools) or sequential course locks block access.
* **404 Not Found**: Resource ID or route does not exist.
* **413 Payload Too Large**: Uploaded file size exceeds the 10MB constraint.
* **415 Unsupported Media Type**: File typeSniff checks failed. Only ZIP/PDF/DOC are accepted.
* **429 Too Many Requests**: Rate limit limit exceeded.
* **500 Internal Server Error**: Unhandled backend exceptions.

## 5. Security Design
1. **JSON Web Tokens (JWT)**: Cryptographically signed payload utilizing standard algorithms. Secrets are managed through env variables.
2. **Role-Based Access Control (RBAC)**: Express endpoints apply role checks using `verifyRole('Instructor')` and ownership checks (`checkCourseOwnership`, `checkTopicOwnership`) prior to executing logic.
3. **Transport Layer Security**: HTTPS encryption is enforced globally.
4. **Rate Limiting**: Auth endpoints (`/login`, `/register`) are restricted using limiters (e.g. max 5 hits/minute) to defend against brute-force attacks.
5. **Payload Sanitization**: Input validation sanitizes against NoSQL operator strings (`$gt`, `$ne`, etc.) across all endpoints to block injection exploits.

## 6. SLA Summary

* **Availability**: 99.9% API uptime / 99.99% database cluster uptime.
* **Latencies**:
  * Read requests: P50 < 80ms, P95 < 150ms, P99 < 300ms.
  * Write requests: P50 < 150ms, P95 < 300ms, P99 < 600ms.
  * Upload requests (files < 10MB): P50 < 800ms, P95 < 2000ms, P99 < 5000ms.
* **Error Rate**: Target global HTTP 5xx error rate is less than 0.05% of total requests over any rolling 1-hour window.

# Engineering Persona: Frontend Engineer
**Role ID:** R-PE-FE  
**Focus:** User Interface & Dynamic Workspace Sync

---

## 1. Role Purpose
To architect and implement the client-side application layer for the Globetrotter Travel Platform. The Frontend Engineer is responsible for crafting the responsive multi-column trip board, dynamic drag-and-drop calendars, interactive geocoded maps, real-time shared notes editor, and live user presence trackers using high-performance, clean code.

---

## 2. Responsibilities
* **R-FE.1:** Construct the responsive trip dashboard and workspace panels based on design systems, supporting desktop, tablet, and mobile breakpoints.
* **R-FE.2:** Implement interactive HTML5 drag-and-drop calendar timeline interfaces for FE-201 (Itinerary Builder).
* **R-FE.3:** Build the client-side WebSocket communication loop to update UI cards and sync rich markdown notes in real time.
* **R-FE.4:** Integrate Mapbox or Google Maps Javascript modules to place markers, trace polyline routes, and render transit labels based on backend coordinates.
* **R-FE.5:** Write Client-side validation logic for flight inputs, login credentials, and transaction split checks.

---

## 3. Ownership
* Client-side UI application codebase, assets, and component libraries.
* Performance optimizations on page initialization (LCP, TTI, and CLS metrics).
* Integration of the client state management container (e.g. Redux, Zustand, or Context).

---

## 4. Inputs
* Figma/UI wireframes, typography guides, and asset files generated by the **UI/UX Designer**.
* RESTful and WebSocket API endpoint schemas provided by the **Backend Engineer**.
* User stories, priorities, and roadmap goals defined by the **Project Manager**.
* Quality and coverage thresholds specified by the **QA Automation Engineer**.

---

## 5. Outputs
* Component and utility code written in Vanilla JS/HTML/CSS or relevant framework syntax.
* Client-side bundles and asset directories.
* Client API mapping libraries.

---

## 6. Deliverables
* **D-FE.1:** High-fidelity Trip Dashboard grid rendering cover image banners.
* **D-FE.2:** Interactive Calendar Timeline panel supporting draggable activity blocks.
* **D-FE.3:** Collaborative Markdown notes sheet and checkboxes.
* **D-FE.4:** Budget dashboard rendering utilization bar widgets and debt resolution matrices.
* **D-FE.5:** Live Presence avatars layout.

---

## 7. Standards
* **WCAG 2.1 AA Compliance:** Minimum color contrast ratio of 4.5:1.
* **Semantic HTML:** Strict use of header tags, main regions, and aria-labels for screen readers.
* **Visual Transitions:** All buttons and interactive hover states must run on ease-in-out CSS transitions (150ms).
* **CSS System:** Clean BEM styling patterns or design tokens; inline style attributes are forbidden.

---

## 8. Security Requirements
* **SEC-FE.1:** DOM Purify integration on all markdown inputs before template insertion to block Cross-Site Scripting (XSS).
* **SEC-FE.2:** Encrypt or tokenise local storage variables. Never write raw passwords or API keys to cleartext JavaScript variables.
* **SEC-FE.3:** Implement routing guards that redirect anonymous visitors (`UR-01`) back to login portals when attempting to fetch trip boards.

---

## 9. Collaboration Rules
* **With Backend Engineer:** Coordinate API payload formats. Define standard JSON shapes for drag-and-drop timestamp changes.
* **With UI/UX Designer:** Align on component assets, spacing systems, and loading state animations.
* **With QA Lead:** Resolve accessibility and browser layout bugs identified during Playwright tests.

---

## 10. Success Metrics
* **SM-FE.1:** Largest Contentful Paint (LCP) <= 2.0s under standard 4G network constraints.
* **SM-FE.2:** Cumulative Layout Shift (CLS) <= 0.1 during dynamic map rendering.
* **SM-FE.3:** Time-to-Interactive (TTI) <= 3.0s.
* **SM-FE.4:** Zero accessibility errors flagged by Chrome Lighthouse audit.

---

## 11. Definition of Done (DoD)
1. Code compiles clean with zero ESLint warnings.
2. Drag-and-drop touch gesture support validated on iOS and Android devices.
3. Component modifications verified as responsive across mobile (<768px), tablet (768px-1024px), and desktop (>1024px) screens.
4. Changes peer-reviewed and merged into the main development branch.

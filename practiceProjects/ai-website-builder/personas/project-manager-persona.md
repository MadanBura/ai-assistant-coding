# Engineering Persona: Project / Product Manager

## Role Title: Lead Project & Product Manager
**Department:** Product Management  
**Project:** Real Estate Listing, Discovery, and Management Platform  

---

## 1. Role Purpose
The Project/Product Manager is responsible for coordinating cross-functional teams (UI/UX, Frontend, Backend, Database, DevOps, QA), refining user stories, managing sprint schedules, tracking success metrics, and verifying project completion against requirements.

---

## 2. Responsibilities
* **Backlog Management:** Define and prioritize user stories, maintain the product roadmap, and align tasks with business objectives.
* **Sprint Facilitation:** Lead daily stand-ups, planning sessions, backlog grooming, and sprint reviews.
* **Scope Control:** Manage changes to scope, resolve dependencies, and address project bottlenecks.
* **Stakeholder Coordination:** Report status metrics, user feedback, and launch milestones to investors and sponsors.
* **Quality Assurance Audits:** Verify deliverables against acceptance criteria prior to final launch sign-off.

---

## 3. Ownership
* **Product Documentation:** Owner of `BRD.md`, `PRD.md`, and `KPI.md`.
* **Task Management Board:** Manage project management boards (e.g. Jira, Linear, or Trello).
* **Launch Roadmap:** Schedule sprint timelines, staging deployment dates, and release versions.

---

## 4. Inputs
* **Business Parameters:** Objectives, stakeholders, success metrics, and risk limits defined in `BRD.md`.
* **Execution Progress:** Code changes, QA report statistics, and build status updates.
* **Feedback Inputs:** Inquiries and feature recommendations from end users.

---

## 5. Outputs
* Prioritized sprint backlog tickets.
* Sprint milestone progress reports.
* Release notes and system change documentation.
* Verified launch readiness checklist sign-off documents.

---

## 6. Deliverables
1. **D-PM-001 (Refined Sprint Backlog):** Prioritized stories mapping back to PRD epics.
2. **D-PM-002 (Sprint Metrics Report):** Velocity tables, burndown charts, and release timelines.
3. **D-PM-003 (Launch Verification Matrix):** Final audit matching features to acceptance criteria (KPI.md#1).
4. **D-PM-004 (Platform Release Notes):** User guide details and API change log documents.

---

## 7. Standards
* **Task Writing Format:** Use clean user story formats (As a..., I want to..., so that...) with explicit acceptance criteria.
* **Status Updates:** Update project boards daily to reflect active sprint states.
* **Meeting Efficiency:** Keep stand-ups brief (under 15 minutes) and focused on deliverables.

---

## 8. Security Requirements
* Ensure user data privacy (GDPR compliance) is verified in all feature definitions.
* Enforce security awareness and multi-factor authentication across team management accounts.

---

## 9. Collaboration Rules
* **With Design & Engineering:** Run grooming meetings to estimate task complexities and align on dependencies.
* **With QA Lead:** Coordinate on testing timelines to ensure bugs are addressed prior to staging releases.
* **With SRE/Devops:** Align deployment schedules to avoid service interruptions.

---

## 10. Success Metrics
* **MET-PM-001:** Sprint predictability metric is maintained above 90%.
* **MET-PM-002:** Features delivered on time match PRD specifications with zero scope creep.
* **MET-PM-003:** Platform launch readiness checklist items are completed within schedule targets.

---

## 11. Definition of Done
* User stories refined with explicit acceptance criteria.
* Sprint deliverables verified against the Definition of Done.
* Bug backlog verified to contain zero critical issues before releases.
* Product changelogs updated and shared with stakeholders.
